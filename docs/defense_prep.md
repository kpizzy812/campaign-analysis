# Подготовка к защите — шпаргалка

## Демо (15 мин)

### Порядок действий:
1. `git clone` + `cp .env.example .env` + вписать ключ + `docker compose up -d`
2. Подождать ~60 сек, проверить `curl http://localhost:8000/health`
3. Отправить запрос к /predict:
```bash
curl -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{"geo":"US","vertical":"gambling","traffic_source":"facebook","device":"mobile","os":"android","bid":0.5,"daily_budget":100,"hour":14,"dow":2}'
```
4. Показать /creatives/analyze:
```bash
curl -X POST http://localhost:8000/creatives/analyze \
  -H "Content-Type: application/json" \
  -d '{"ad_text":"Получи 200% бонус! 50,000+ игроков уже выиграли. Только 24 часа!"}'
```
5. Показать /creatives/generate:
```bash
curl -X POST http://localhost:8000/creatives/generate \
  -H "Content-Type: application/json" \
  -d '{"offer":"Казино с бонусом 200%","geo":"US","vertical":"gambling","n_variants":3}'
```
6. Показать /health с cache stats и /metrics

## Технические вопросы (15 мин)

### «Почему LightGBM а не XGBoost?»

Оба — gradient boosting. Конкретные trade-offs:
- **LightGBM быстрее** на обучении: leaf-wise growth vs level-wise у XGBoost. На наших 5K строках разница небольшая, но при масштабировании до миллионов строк — в 2-5x быстрее
- **LightGBM нативно работает с категориальными фичами** — OrdinalEncoder достаточно. XGBoost требует OneHot
- **Inference скорость сравнима** — оба <10ms на одно предсказание
- **XGBoost преимущество:** лучше на маленьких датасетах, менее склонен к переобучению. Для наших 5K строк разница минимальна
- **Вывод:** для первой версии разница несущественна. Выбрал LightGBM по скорости обучения (Optuna 30 trials = 150 fit-predict циклов)

### «Модель вернула 0.51 — запускать или нет?»

0.51 — зона неопределённости. Ответ зависит от бизнес-контекста:
- **Default threshold = 0.5** — математически оптимально для balanced accuracy
- **Но для MST важнее precision чем recall** — лучше пропустить хорошую кампанию, чем запустить убыточную (деньги теряем сразу)
- **Рекомендация: поднять threshold до 0.65-0.70.** При 0.70: precision вырастет до ~98%, но recall упадёт до ~75%. Пропустим больше кампаний, но те что запустим — почти гарантированно прибыльные
- **0.51 при threshold 0.70 = не запускать.** Модель не уверена, лучше не рисковать
- **Можно сделать 3 зоны:** <0.4 — точно нет, >0.7 — точно да, 0.4-0.7 — решение байера

### «Через неделю метрики упали — первые три шага»

1. **Проверить data drift:** сравнить распределение фичей в свежих данных vs training data. Если появились новые гео, вертикали, аномальные биды — модель видит то, чему не обучалась
2. **Проверить target drift:** средний ROI кампаний за неделю vs исторический. Если рынок сместился (например, payout снизился у оффера) — модель не причём, данные изменились
3. **Переобучить на свежих данных:** добавить последние 2 недели в training set, перезапустить Optuna, сравнить метрики. Если лучше — деплоить. Если нет — копать глубже в причину drift

### «Байер говорит модель ошибается на push-трафике из Индии»

1. **Проверить данные:** сколько кампаний push/IN было в training? Если <20 — модель просто не видела достаточно примеров этой комбинации
2. **Посмотреть actual vs predicted:** на тех кампаниях push/IN которые есть — модель систематически завышает или занижает? Это bias для конкретного сегмента
3. **Решение:** если мало данных — добавить interaction feature `source_geo` = "push_IN" и набрать больше данных. Если данных достаточно но модель ошибается — возможно для IN другая модель нужна (пайплайн по гео-тирам)
4. **Временный fix:** для push/IN добавить корректирующий коэффициент на основе фактических данных байера

### «Нужно добавить новую вертикаль — как меняешь пайплайн?»

1. **Данные:** нужно минимум 200-300 завершённых кампаний в новой вертикали с известным ROI
2. **Код:** ничего менять не надо. `vertical` — это категориальная фича через OrdinalEncoder. Новое значение просто появится в данных
3. **Переобучение:** добавить данные в train → перезапустить `python -m src.train` → Optuna подберёт параметры заново
4. **Encoder:** OrdinalEncoder с `handle_unknown='use_encoded_value'` — новые категории не ломают inference. При переобучении encoder учит новую вертикаль
5. **Риск:** если новая вертикаль сильно отличается от существующих (например fintech vs gambling), модель может плохо генерализировать. Мониторить первые 2 недели

## Ситуационные задачи (15 мин)

### Задача A: 3 ночи, error rate 15%, Redis недоступен

**Прямо сейчас (0-5 мин):**
1. API продолжает работать — в коде есть fallback на прямой вызов модели без кэша (`if cached_predictor is None` → direct model call). Error rate 15% = скорее всего таймауты на Redis connection, а не полный отказ
2. Проверить: `redis-cli ping` — если Redis упал, смотреть `systemctl status redis` и `redis-cli info memory` (OOM?)
3. Если Redis OOM — `redis-cli FLUSHDB` очистить кэш (предсказания пересчитаются), увеличить maxmemory

**Восстановление (5-30 мин):**
4. Перезапустить Redis: `systemctl restart redis`
5. Проверить что API перестал генерировать ошибки
6. Утром: настроить Redis maxmemory-policy = allkeys-lru (автоочистка старых ключей), добавить alert на memory > 80%

### Задача B: Добавить предсказание LTV (regression)

**Изменения в пайплайне:**
1. **Таргет:** вместо `is_profitable` (0/1) → `revenue` (float). Регрессия вместо классификации
2. **Модель:** `LGBMRegressor` вместо `LGBMClassifier`. Тот же pipeline, другой estimator
3. **Метрики:** MAE, RMSE, R² вместо ROC-AUC. Бизнес-метрика: средняя ошибка предсказания в долларах
4. **Optuna:** те же 6 параметров, но `scoring='neg_mean_absolute_error'`
5. **API:** новый endpoint `POST /predict/ltv` → возвращает `{"predicted_revenue": 150.5, "confidence_interval": [80, 220]}`. Существующий `/predict` остаётся как есть
6. **Риск:** LTV имеет тяжёлый правый хвост (редкие whale-кампании). Нужно log-transform таргета или quantile regression

### Задача C: Объяснить байеру почему модель не рекомендовала кампанию

**Без технического жаргона:**

"Модель посмотрела на параметры твоей кампании и сравнила с 5,000 похожих кампаний из прошлого. Вот что повлияло:

1. **Гео IN (Индия)** — из похожих кампаний в Индии только 25% были прибыльными. Это самый слабый гео в нашей базе
2. **Push-трафик** — CTR хороший, но конверсия в Индии через push низкая
3. **Бид $0.05** — при таком биде мы получаем самый дешёвый трафик, но он хуже конвертится

Это не значит что кампания точно провалится. Модель говорит что из 10 таких кампаний прибыльными были 2-3. Если у тебя есть основания думать что этот оффер особенный — попробуй на маленьком бюджете."

**Техническая реализация:** SHAP values для feature importance per prediction. `shap.TreeExplainer(model).shap_values(X)` → визуализация вклада каждой фичи.
