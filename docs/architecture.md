# Production Architecture: Real-time Inference Pipeline

## Поток данных

```
Событие (клик/показ)
    → Kafka (очередь событий)
    → Feature Store (Redis)
    → Model Server (FastAPI + uvicorn, 4 workers)
    → Предсказание
    → Redis (кэш предсказаний)
```

| Компонент | Технология | Почему |
|-----------|-----------|--------|
| Очередь событий | **Kafka** | Буферизует 10M+ событий/день, гарантирует at-least-once delivery, позволяет replay при сбоях |
| Feature Store | **Redis** | Latency <1ms на чтение, TTL для автообновления, атомарные операции для счётчиков |
| Model Server | **FastAPI + uvicorn** | Async I/O, multi-worker (prefork), P95 < 50ms на нашем бенчмарке |
| Response Cache | **Redis** | Тот же инстанс, отдельный DB namespace. Снижает нагрузку на модель в 5-10x через cache hit |
| Мониторинг | **Prometheus + Grafana** | Метрики latency, RPS, error rate, cache hit rate в реальном времени |

## Feature Store

| Признак | Частота обновления | Хранилище | Почему |
|---------|-------------------|-----------|--------|
| CTR по гео за 7 дней | Каждый час | Redis (TTL 2h) | Часовое обновление балансирует свежесть данных и нагрузку на расчёт. TTL 2h страхует от пропуска джобы |
| Статус кампании (активна/пауза) | Real-time (event-driven) | Redis (no TTL) | Статус меняется мгновенно — байер нажал «пауза», модель должна это увидеть на следующем запросе |
| Сезонный коэффициент по часу | Ежедневно | Redis (TTL 48h) | Сезонность меняется медленно. Пересчёт ночью по данным за 30 дней |

## Модель в продакшене

### Batch vs Online — выбор: Online inference

Online inference, не batch. Причины:
- Требование latency <100ms исключает batch (batch = задержка минимум N минут)
- Решение «показывать объявление или нет» принимается в реальном времени
- С Redis-кэшем (hit rate ~97% на нашем тесте) реальная нагрузка на модель = 3% от трафика

### Rollback стратегия

Blue-green deploy через nginx upstream:
1. Новая версия поднимается на порту 8081 (green), старая на 8080 (blue)
2. Smoke test: 100 запросов на green, проверка метрик
3. Nginx переключает трафик на green
4. Если CTR падает >10% за 15 минут — автоматический rollback на blue через Prometheus alert → nginx reload
5. Blue остаётся живым 1 час после переключения

### Расчёт реплик

Данные из нагрузочного теста (Linux, 4 workers, Redis cache):
- P50 = 42ms, P95 = 44ms при 120 RPS
- Один инстанс (4 workers) держит 120 RPS с запасом

Для 115 RPS (10M событий/день):
- **2 реплики** — одна active, одна standby (failover)
- При росте до 500 RPS: горизонтальное масштабирование через k8s HPA по CPU/latency
- Каждая реплика: 4 uvicorn workers, 2 vCPU, 1GB RAM

## Три сценария отказа

### 1. Модель перестала отвечать (первые 5 минут)

1. **0-30 сек:** Prometheus alert на error rate >5%. PagerDuty уведомляет дежурного
2. **30 сек - 2 мин:** Nginx health check отключает больной инстанс, трафик уходит на standby реплику. Кэшированные предсказания продолжают отдаваться из Redis
3. **2-5 мин:** Дежурный проверяет логи (`kubectl logs`), определяет причину: OOM → увеличить memory limit и restart; deadlock → restart pod; corrupted model file → rollback на предыдущую версию через `kubectl rollout undo`
4. Если причина не ясна за 5 минут — fallback на default prediction (средний CTR по вертикали из Feature Store)

### 2. CTR упал на 50% после деплоя новой модели

1. **Rollback немедленно:** переключить nginx обратно на blue (старую модель). Не тратить время на диагностику — сначала остановить потери
2. **Диагностика:** сравнить feature distributions train vs production (data drift?). Проверить: новые категории в гео/вертикали которых не было в train? Scaler/encoder обучен на старых данных?
3. **Fix forward:** пересобрать модель с актуальными данными, прогнать A/B тест на 5% трафика перед полным деплоем

### 3. Feature Store вернул null для 30% запросов

Fallback-стратегия по приоритету:
1. **Первый уровень:** использовать значение из локального in-memory кэша (последнее известное значение фичи). LRU-кэш на 10K ключей в каждом worker
2. **Второй уровень:** подставить default значение — медиана фичи по train набору. Defaults хранятся в `models/metadata.json` и загружаются при старте
3. **Третий уровень:** если >50% фичей = null, вернуть fallback prediction (средний CTR по вертикали) с флагом `"degraded": true` в ответе
4. **Параллельно:** alert дежурному. Причина null обычно: Redis OOM (увеличить maxmemory), сетевой сбой (проверить connectivity), джоба обновления фичей упала (перезапустить Airflow DAG)
